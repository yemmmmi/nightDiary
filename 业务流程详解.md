# 夜记 AI 系统 — 业务流程详解

本文档详细解释从用户输入到系统输出的完整业务流程，涵盖所有涉及的模块及其工作原理。

---

## 一、全局流程概览

```
用户写日记 → 保存日记 → 触发 AI 分析 → 展示 AI 回应 → 用户反馈 → 系统学习
    │            │            │              │             │           │
    ▼            ▼            ▼              ▼             ▼           ▼
 DiaryEditor  DiaryService  AIService    AIAnalysis   Feedback    Thompson
 (前端)       (后端)        (路由决策)    Panel(前端)   Buttons    Sampling
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
              Multi-Agent   ReAct Agent   Chain
              (LangGraph)   (工具调用)    (直接生成)
```

---

## 二、详细流程分解

### 阶段 1：用户输入（前端）

**涉及组件：** `DiaryEditor.vue` → `DiaryPage.vue`

1. 用户在 `DiaryEditor` 中输入日记内容
2. 可选择标签（Few-shot 上下文）、设置公开/私密
3. 点击"发布日记"，调用 `POST /diary/entries`
4. 日记保存成功后，前端刷新日记列表

**关键细节：**
- 编辑器监听内容变化，防抖 1 秒后发射 `editing` 事件（用于 inspired_writing 隐式反馈）
- 日记保存时自动获取天气信息（高德地图 API）
- 日记内容同步写入 Chroma 向量库（用于后续 RAG 检索）

---

### 阶段 2：触发 AI 分析

**涉及组件：** `AIAnalysisPanel.vue` → `POST /analysis` → `AnalysisService` → `AIService`

1. 用户点击日记条目，右侧面板显示 `AIAnalysisPanel`
2. 用户点击"获取 AI 总结"按钮
3. 前端调用 `POST /analysis`（传入 diary NID）
4. 后端 `AnalysisService` 检查防重（同一日记内容未变化不重复分析）
5. 调用 `AIService.analyze()` 进入核心 AI 流程

**隐式反馈触发点：**
- 点击分析按钮时，`checkFrequentUsage()` 检查是否 24 小时内再次触发（frequent_usage 信号）

---

### 阶段 3：AIService 路由决策

**涉及组件：** `backend/app/services/ai_service.py`

`AIService.analyze()` 是整个 AI 模块的入口，它根据条件选择执行模式：

```python
# 路由优先级
if _should_use_multi_agent(db, user_id):    # 条件：langgraph 可用 + 功能开关 + db/user_id
    result = _run_multi_agent(...)           # → Multi-Agent 流程
elif db and user_id and has_temporal_ref:    # 条件：有时间回溯关键词
    result = _run_agent(...)                 # → ReAct Agent 流程
else:
    result = _run_chain(...)                 # → 简单 Chain 流程
```

**路由条件：**
| 模式 | 触发条件 | 特点 |
|------|----------|------|
| Multi-Agent | langgraph 已安装 + `MULTI_AGENT_ENABLED=true` + db/user_id 提供 | 最强大，多 Agent 协作 |
| ReAct Agent | db/user_id 提供 + 日记含时间回溯关键词（"昨天"、"之前"等） | 支持工具调用 |
| Chain | 其他情况 | 最简单，直接 Prompt → LLM |

---

### 阶段 4：Multi-Agent 流程（核心）

**涉及组件：** `agents/` 目录下所有模块

这是系统最核心的流程，通过 LangGraph 状态图协调多个 Agent 协作：

#### Step 4.1：加载记忆上下文

```
┌─────────────────────────────────────────┐
│ 1. Episodic Memory (Redis)              │
│    → retrieve_relevant(user_id, query)  │
│    → 返回 top 5 相关历史交互记忆         │
│                                         │
│ 2. Long-Term Memory (MySQL)             │
│    → get_profile(user_id)               │
│    → 返回用户画像（性格、情绪基线、偏好） │
└─────────────────────────────────────────┘
```

**Episodic Memory 工作原理：**
- 存储在 Redis Sorted Set 中，key = `memory:episodic:{user_id}`
- 每条记忆有 importance 分数（0-1），按 importance × 时间衰减因子排序
- 超过 100 条时自动驱逐最低重要性的条目
- Redis 不可用时优雅降级（返回空列表）

**Long-Term Memory 工作原理：**
- 存储在 User 表的 `long_term_profile` JSON 字段中
- 包含：personality_tags, emotion_baseline, important_people, recurring_topics, preferred_response_style
- 连续 3 天出现的情绪主题/话题会被提升到画像中

#### Step 4.2：构建初始状态

```python
initial_state = {
    "diary_content": "今天工作压力很大...",
    "user_id": 1,
    "diary_nid": 100,
    "episodic_context": [...],      # 从 Episodic Memory 加载
    "long_term_profile": {...},     # 从 Long-Term Memory 加载
    "intent": "",                   # 待 Supervisor 填充
    "token_budget": 0,              # 待 Supervisor 分配
    "activated_agents": [],         # 待 Supervisor 决定
    "retrieval_context": "",        # 待 Retrieval Agent 填充
    "empathy_response": "",         # 待 Empathy Agent 填充
    "insight_response": "",         # 待 Insight Agent 填充
    "final_response": "",           # 待 Supervisor 整合
    "errors": [],                   # Worker 失败时记录
}
```

#### Step 4.3：Supervisor Agent — 意图分类

**涉及组件：** `agents/supervisor.py` + `agents/intent_classifier.py`

```
日记内容 → IntentClassifier
              │
              ├── 规则层（置信度 > 0.9 直接返回）
              │   • "记录一下今天吃了面条" → pure_record (0.95)
              │   • "今天真的太崩溃了" → emotional_support (0.92)
              │   • "和之前写过的一样" → retrospective_review (0.91)
              │
              └── LLM 层（模糊场景，单次调用 ≤ 100 tokens）
                  • 输出 IntentResult(need_retrieval, need_weather, need_analysis, confidence)
```

**分类后的动作：**
| 意图 | Token 预算 | 激活的 Worker Agents |
|------|-----------|---------------------|
| pure_record | 400-600 | Empathy |
| emotional_support | 1000-1500 | Empathy + Retrieval |
| retrospective_review | 1500-2500 | Empathy + Retrieval + Insight |
| habit_tracking | 1200-2000 | Retrieval + Insight |

#### Step 4.4：条件路由 → Worker Agents 并行执行

LangGraph 的条件路由器根据 `state["intent"]` 将执行分发到对应的 Worker Agent 节点。多个 Worker 可以并行执行。

### Worker Agent 详解

#### Empathy Agent（情感共鸣）

**文件：** `agents/empathy_agent.py`

**职责：** 生成确认用户情绪的温暖回应，是所有意图都会激活的核心 Agent。

**执行流程：**
```
1. 获取用户偏好回应风格（从 long_term_profile.preferred_response_style）
2. 情绪评估：
   - 从画像获取 emotion_baseline
   - 从日记内容关键词估算当前情绪分数
   - 取两者中更负面的值
3. 危机检测：情绪分数 < -0.7 → 触发危机响应路径
4. 格式化 Episodic Memory 上下文（最多 5 条，体现关怀连续性）
5. 查询 Domain Knowledge Store（仅 emotional_support/retrospective_review 或危机时）
6. 注入 PromptTuner 动态偏好片段（风格 + 长度 + 直接度）
7. 构建系统 Prompt → 调用 LLM 生成回应
8. 危机模式：在回应后附加心理援助热线信息
```

**回应长度控制：**
| 意图 | 汉字数 |
|------|--------|
| pure_record | 50-150 |
| emotional_support | 100-300 |

**降级行为：** LLM 失败时返回预设的降级文本（危机模式下仍包含热线信息）

---

#### Retrieval Agent（RAG 增强检索）

**文件：** `agents/retrieval_agent.py`

**职责：** 从用户历史日记中检索相关信息，生成结构化摘要供其他 Agent 使用。

**执行流程：**
```
1. 时间范围推断：
   - 解析日记中的时间表达（"昨天"→1天, "上周"→7天, "3个月前"→90天）
   - 支持具体日期格式（2024年12月25日）
   
2. 多跳检索（最多 3 次）：
   - 第 1 跳：使用原始日记内容作为查询
   - 评估结果相关性（rerank_score / rrf_score）
   - 相关性 < 0.3 且结果不足 → 生成改进查询 → 第 2 跳
   - 每跳去重合并结果
   
3. 混合检索管线：
   - Chroma 向量检索（text2vec-base-chinese Embedding）
   - BM25 关键词检索
   - RRF（Reciprocal Rank Fusion）融合排序
   - bge-reranker-base 重排序
   
4. 查询结构化知识库（KnowledgeEntry 表，关键词匹配）
5. 查询 Domain Knowledge Store（Chroma 共享集合，语义检索）
6. 生成结构化摘要（≤ 300 tokens）：
   - 【相关日记】历史日记片段
   - 【知识关联】结构化实体信息
   - 【领域参考】心理学专业知识
```

**降级行为：** 任何步骤失败都不会阻断流程，返回已获取的部分结果

---

#### Insight Agent（洞察分析）

**文件：** `agents/insight_agent.py`

**职责：** 分析情绪模式和行为趋势，生成可操作建议或结构化报告。

**执行流程：**
```
1. 检测是否为周报/月报请求（关键词匹配）
2. 解析 emotion_baseline（从 long_term_profile）
3. 情绪偏离检测：
   - 从 episodic_context 提取近期情绪信号
   - 计算近期情绪均值 vs 基线均值
   - 偏离 ≥ 0.3 → 标记为显著偏离
4. 构建上下文摘要（retrieval_context + episodic_context + long_term_profile）
5. 查询 Domain Knowledge Store
6. 注入 PromptTuner 动态偏好片段
7. 选择系统 Prompt：
   - 周报/月报模式 → 结构化报告 Prompt（主导情绪/关键事件/趋势/建议）
   - 常规模式 → 洞察分析 Prompt
8. 调用 LLM 生成洞察
```

**情绪偏离检测逻辑：**
```python
# 从 episodic_context 中提取情绪极性
recent_emotions = [map_emotion_to_score(entry["emotion"]) for entry in episodic_context]
recent_avg = mean(recent_emotions)
deviation = recent_avg - baseline.average_sentiment
if abs(deviation) >= 0.3:
    # 在 LLM Prompt 中加入偏离提醒
```

---

#### Step 4.5：Supervisor Agent — 结果整合

**涉及组件：** `agents/supervisor.py` 的 `synthesize_response()` 方法

```
收集各 Worker 输出：
  - empathy_response: "理解你今天的压力..."
  - retrieval_context: "【相关日记】上周也提到了加班..."
  - insight_response: "从最近的模式来看，你的焦虑与工作节奏相关..."

整合策略：
  ├── 有 LLM 且多个输出 → LLM 融合为自然连贯的回应
  └── 无 LLM 或单个输出 → 按优先级拼接（empathy > insight > retrieval）

Worker 失败处理：
  - 部分失败 → 使用已成功的输出生成回应
  - 全部失败 → 返回 FALLBACK_FEEDBACK 兜底文本
```

---

### 阶段 5：结果存储与返回

**涉及组件：** `services/analysis_service.py` + `models/analysis.py`

1. AI 回应写入 `DiaryEntry.AI_ans` 字段
2. 创建 Analysis 记录：
   - `Token_cost`：总 Token 消耗
   - `cache_hit_tokens`：DeepSeek 缓存命中（免费）
   - `cache_miss_tokens`：付费输入 Token
   - `output_tokens`：付费输出 Token
   - `agent_mode`：使用的模式（chain/agent/multi_agent）
   - `activated_agents`：激活的 Worker 列表
   - `Thk_log`：执行日志
3. 异步触发知识抽取（`KnowledgeExtractor`，日记 > 100 字符时）
4. 异步触发 Episodic Memory 写入（`MemoryWriterSkill`，重要性 > 0.5 时）

---

### 阶段 6：前端展示 AI 回应

**涉及组件：** `AIAnalysisPanel.vue` + `FeedbackButtons.vue`

1. 前端收到分析结果，渲染 AI 回应文本
2. 显示 Token 消耗明细（可展开）
3. 显示 Agent 模式标签（Chain / Agent / Multi-Agent）
4. 触发 `observeReadComplete()`：IntersectionObserver 监测回应元素可见度
5. 触发 `markAiResponseReceived()`：开始 5 分钟 inspired_writing 计时窗口

---

### 阶段 7：用户反馈与系统学习

**涉及组件：** `FeedbackButtons.vue` → `POST /api/feedback` → `ThompsonSampling`

#### 显式反馈流程：
```
用户点击 👍 → POST /api/feedback {feedback_type: "positive", response_style: "empathetic"}
                    │
                    ├── 持久化到 Feedback 表
                    └── BackgroundTask: ThompsonSampling.update_reward(user_id, "empathetic", is_positive=True)
                                            │
                                            └── StylePreference.alpha += 1  (empathetic 风格的正向证据增加)

用户点击 👎 → 显示原因选择器 → 选择"太笼统"
         → POST /api/feedback {feedback_type: "negative", reason: "too_generic", response_style: "empathetic"}
                    │
                    ├── 持久化到 Feedback 表
                    └── BackgroundTask: ThompsonSampling.update_reward(user_id, "empathetic", is_positive=False)
                                            │
                                            └── StylePreference.beta += 1  (empathetic 风格的负向证据增加)
```

#### 隐式反馈流程：
```
read_complete:     用户完整阅读 AI 回应（元素 90%+ 可见）→ 正向信号
inspired_writing:  收到回应后 5 分钟内继续编辑日记 → 正向信号
frequent_usage:    24 小时内再次触发分析 → 正向信号
```

#### Thompson Sampling 如何影响下次回应：
```
下次分析请求 → PromptTuner.get_user_preference(user_id)
                    │
                    ├── 读取 StylePreference 表（4 种风格的 alpha, beta 值）
                    ├── 对每种风格的 Beta(alpha, beta) 分布采样
                    ├── 选择采样值最高的风格
                    ├── 推断 response_length 和 directness
                    └── 构建动态 Prompt 片段注入到 Empathy/Insight Agent
```

**示例：** 用户多次点赞 practical 风格的回应后：
- practical: Beta(8, 2) → 采样均值约 0.8
- empathetic: Beta(3, 5) → 采样均值约 0.375
- 系统更倾向选择 practical 风格

## 三、辅助流程

### 3.1 知识抽取流程（异步，不阻塞主流程）

**触发条件：** 日记内容 > 100 字符

```
日记保存成功 → fire_and_forget_extraction(user_id, diary_nid, content)
                    │
                    ├── 构建结构化输出 Prompt（单次 LLM 调用，≤ 500 tokens）
                    ├── LLM 返回 JSON：{persons: [], events: [], places: [], topics: [], mood_score: 0.5}
                    ├── 解析 JSON → 为每个实体创建 KnowledgeEntry 记录
                    └── 存储到 MySQL（关联 user_id + diary_nid）
                    
失败时：静默跳过，不影响日记保存
```

**抽取的实体类型：**
| 类型 | 示例 |
|------|------|
| person | {"name": "小明", "relation": "同事"} |
| event | {"description": "项目截止", "date": "2024-01-15"} |
| place | {"name": "公司", "type": "workplace"} |
| topic | {"name": "工作压力", "sentiment": "negative"} |
| mood | {"score": -0.3, "keywords": ["焦虑", "压力"]} |

### 3.3 PromptTuner 动态 Prompt 注入

**每次 AI 分析时实时执行（无缓存，偏好变化立即生效）：**

```
PromptTuner.build_dynamic_prompt(user_id, "empathy")
    │
    ├── 查询 StylePreference 表
    ├── Thompson Sampling 采样选择风格
    ├── 推断 response_length（short/medium/long）
    ├── 推断 directness（0.0-1.0）
    └── 生成 Prompt 片段：
        
        ## 用户偏好适配指令
        
        ### 回应风格
        务实关怀、在表达理解的同时给出具体可操作的建议。语言简洁有力。
        
        ### 回应长度
        请将回应控制在适中长度（100-200 字），平衡深度和简洁。
        
        ### 表达直接度
        语言风格偏直接坦率，清晰指出观察到的问题和建议，不绕弯子。
```

## 五、关键设计决策

| 决策 | 原因 |
|------|------|
| LangGraph 而非自定义调度 | 原生支持条件路由、并行执行、状态管理，减少自研代码 |
| Annotated TypedDict + reducer | 解决多 Worker 并发写入同一 state 字段的冲突问题 |
| 规则层 + LLM 层两级分类 | 高置信度场景跳过 LLM 调用，节省 Token 和延迟 |
| Redis Sorted Set 存储 Episodic Memory | 天然支持按分数排序、范围查询、自动驱逐 |
| Thompson Sampling 而非 UCB/ε-greedy | 自然平衡探索与利用，新用户均匀先验无需冷启动 |
| BackgroundTasks 异步更新 Beta 分布 | 确保反馈 API 200ms 内响应，不阻塞 UI |
| 每次请求实时读取偏好（无缓存） | 偏好变化下次请求立即生效，简化一致性问题 |
| Worker 失败记录到 errors 而非抛异常 | 允许 Supervisor 使用部分成功的输出，提高系统韧性 |
| Domain Knowledge Store 共享集合 | 心理学知识对所有用户通用，无需按用户隔离 |
| 所有数据查询强制 user_id 过滤 | 严格数据隔离，防止跨用户数据泄露 |

## 七、总结

夜记 AI 系统的核心价值在于：

1. **个性化**：通过三层记忆系统和 Thompson Sampling，系统能记住用户的情绪模式、重要人物和偏好风格，提供越来越贴合的回应
2. **智能路由**：不同类型的日记获得不同深度的分析，纯记录不浪费 Token，情感支持获得充分关怀
3. **韧性设计**：任何单一组件故障都不会导致系统不可用，层层降级确保用户始终能获得回应
4. **持续学习**：显式和隐式反馈信号持续优化系统行为，无需人工干预
5. **专业支撑**：85 条精选心理学知识为 AI 回应提供专业基础，危机检测保障用户安全
