## 一、Multi-Agent 架构升级

### 1.1 现状分析

当前系统是单 Agent 架构：一个 ReAct Agent 绑定 4 个工具（search_diary, get_weather_info, analyze_sentiment, get_user_address），根据关键词决定是否启用 Agent 模式。

**痛点：**
- 单 Agent 承担所有职责（情感陪伴 + 信息检索 + 分析总结），Prompt 膨胀导致指令遵从率下降
- 工具调用决策完全依赖 LLM 自行判断，偶发"死循环"（mythink.txt 中提到的 Agent 死机问题）
- 无法并行处理复杂任务（如同时检索历史 + 分析情感 + 生成建议）

### 1.2 优化方案：Supervisor + Worker 多 Agent 架构

```
┌─────────────────────────────────────────────────────┐
│                   Supervisor Agent                    │
│  职责：意图识别、任务拆解、结果整合、最终回应生成       │
└──────────┬──────────────┬──────────────┬────────────┘
           │              │              │
    ┌──────▼──────┐ ┌────▼─────┐ ┌─────▼──────┐
    │ Retrieval   │ │ Empathy  │ │ Insight    │
    │ Agent       │ │ Agent    │ │ Agent      │
    │             │ │          │ │            │
    │ - RAG 检索  │ │ - 情感共鸣│ │ - 模式发现 │
    │ - 天气查询  │ │ - 陪伴对话│ │ - 趋势分析 │
    │ - 事实核查  │ │ - 危机识别│ │ - 行为建议 │
    └─────────────┘ └──────────┘ └────────────┘
```

**Supervisor Agent（协调者）：**
- 接收用户日记，进行意图分类（纯记录 / 求助倾诉 / 回顾反思 / 习惯追踪）
- 根据意图决定激活哪些 Worker Agent（轻量日记只激活 Empathy，复杂回顾同时激活三个）
- 整合多 Agent 输出，生成统一回应
- 控制总 Token 预算，避免超支

**Retrieval Agent（检索者）：**
- 专注 RAG 检索、天气查询、地址获取
- 拥有更精细的检索策略（时间范围推断、多跳检索）
- 负责将检索结果结构化摘要后传递给其他 Agent

**Empathy Agent（共情者）：**
- 专注情感陪伴和心理支持
- 接收情感分析结果，生成温暖、有共情力的回应
- 内置危机检测（识别极端负面情绪，触发特殊回应路径）

**Insight Agent（洞察者）：**
- 专注长期模式发现（情绪趋势、行为规律、习惯变化）
- 跨日记关联分析（本周 vs 上周情绪对比）
- 生成可操作的建议（而非泛泛鸡汤）

## 二、Memory 系统设计

### 2.1 现状分析

当前"记忆"能力有限：
- 短期：仅注入最近 7 天日记摘要（`get_recent_7days`），固定窗口，无优先级
- 长期：完全依赖 RAG 语义检索，只有用户主动提及历史时才触发
- 无工作记忆：Agent 不记得上一次分析了什么、给了什么建议

### 2.2 三层 Memory 架构

```
┌───────────────────────────────────────────────────┐
│  Layer 3: Long-term Memory (知识图谱 / 用户画像)    │
│  存储：Neo4j / JSON 文件                           │
│  内容：用户性格标签、长期情绪基线、重要人物关系       │
│  更新频率：每周/每月自动更新                         │
├───────────────────────────────────────────────────┤
│  Layer 2: Episodic Memory (情景记忆)                │
│  存储：Redis Sorted Set                            │
│  内容：近期重要事件摘要、AI 上次建议、用户反馈        │
│  更新频率：每次交互后更新                            │
├───────────────────────────────────────────────────┤
│  Layer 1: Working Memory (工作记忆)                 │
│  存储：内存 / Prompt Context Window                 │
│  内容：当前会话上下文、本次任务中间状态              │
│  生命周期：单次分析请求内                            │
└───────────────────────────────────────────────────┘
```

### 2.3 Episodic Memory 详细设计

```python
# Redis 数据结构

# Type: Sorted Set (score = timestamp)
# Member: JSON string

episodic_entry = {
    "event": "用户连续三天提到加班压力",
    "emotion": "焦虑",
    "ai_suggestion": "建议设定工作时间边界",
    "user_feedback": None,  # 用户是否采纳（未来可通过交互获取）
    "timestamp": 1716000000,
    "related_nids": [101, 103, 105],
    "importance": 0.8,  # 重要性评分（由 LLM 判定）
}
```

**Memory 写入时机：**
- 每次 AI 分析完成后，Supervisor Agent 判断本次交互是否产生"值得记住"的信息
- 重要性评分 > 0.5 的事件写入 Episodic Memory
- 连续 3+ 天出现相同情绪/主题时，自动升级为 Long-term Memory

**Memory 读取策略：**
- 每次分析前，从 Episodic Memory 取最近 5 条（按 importance × recency 排序）
- 注入 Prompt 作为"AI 的记忆"，让回应具有连续性

## 三、RL（强化学习）系统

### 3.1 现状分析

当前 AI 回应质量无反馈闭环：
- 用户无法对 AI 回应进行评价
- 无法根据用户偏好调整回应风格
- Prompt 中的指令是静态的，不会随使用而进化

### 3.2 RLHF-Lite: 基于用户反馈的轻量强化学习

由于日记场景不适合做全量 RLHF（数据量小、无法训练模型），采用 **Prompt 级别的强化学习**：

```
┌─────────┐     ┌──────────┐     ┌───────────────┐
│ AI 回应  │ ──→ │ 用户反馈  │ ──→ │ Reward Model  │
│          │     │ 👍 / 👎  │     │ (规则 + LLM)   │
└─────────┘     └──────────┘     └───────┬───────┘
                                          │
                                          ▼
                                  ┌───────────────┐
                                  │ Prompt Tuner  │
                                  │ 动态调整策略   │
                                  └───────────────┘
```

### 3.3 反馈收集机制

**显式反馈：**
- AI 分析结果旁添加 👍/👎 按钮
- 可选：用户点击 👎 后弹出原因选择（太长/太短/不相关/太鸡汤/缺乏建议）

**隐式反馈：**
- 用户是否阅读了完整回应（前端追踪展开/收起行为）
- 用户在 AI 回应后是否继续编辑日记（表示被激发表达欲 → 正向信号）
- 用户是否在短期内再次使用 AI 分析（高频使用 → 满意度高）

### 3.4 Reward Signal → Prompt 动态调整

```python
# 用户偏好向量（从反馈中学习）
user_preference = {
    "response_length": "short",      # short / medium / long
    "style": "practical",            # practical / empathetic / philosophical
    "tool_usage": "minimal",         # minimal / moderate / aggressive
    "directness": 0.7,               # 0=委婉 1=直接
}

# 动态 Prompt 片段注入
def build_dynamic_prompt(user_preference: dict) -> str:
    fragments = []
    if user_preference["response_length"] == "short":
        fragments.append("回应控制在 50 字以内。")
    if user_preference["style"] == "practical":
        fragments.append("侧重具体可操作的建议，避免空泛安慰。")
    if user_preference["directness"] > 0.6:
        fragments.append("直接指出问题，不用过多铺垫。")
    return "\n".join(fragments)
```

### 3.5 Multi-Armed Bandit 策略选择

对于回应风格，使用 Thompson Sampling 探索最优策略：

```python
class ResponseStyleBandit:
    """
    为每个用户维护多个回应风格的 Beta 分布参数。
    每次选择一个风格，根据用户反馈更新分布。
    """
    styles = ["empathetic", "practical", "philosophical", "humorous"]
    
    def __init__(self, user_id: int):
        # alpha=成功次数+1, beta=失败次数+1 (Beta 分布先验)
        self.params = {s: {"alpha": 1, "beta": 1} for s in self.styles}
    
    def select_style(self) -> str:
        """Thompson Sampling: 从每个 arm 的 Beta 分布采样，选最大值"""
        samples = {
            s: np.random.beta(p["alpha"], p["beta"])
            for s, p in self.params.items()
        }
        return max(samples, key=samples.get)
    
    def update(self, style: str, reward: bool):
        """根据用户反馈更新分布"""
        if reward:
            self.params[style]["alpha"] += 1
        else:
            self.params[style]["beta"] += 1
```

## 四、上下文处理优化

### 4.1 现状分析

当前上下文构建策略较为粗糙：
- 历史日记：固定取最近 7 天全部内容，无优先级排序
- 标签上下文：简单拼接所有标签名
- 天气信息：固定注入当前天气字符串
- Agent 模式：依赖 25 个硬编码时间关键词触发

**问题：**
- 7 天窗口内日记过多时 Token 浪费严重
- 无关日记（如纯流水账）和高相关日记（情绪相近）被同等对待
- 上下文窗口没有根据模型 max_tokens 做自适应裁剪

### 4.2 智能上下文压缩

```python
class ContextCompressor:
    """
    根据当前日记内容，动态选择最相关的历史上下文。
    而非固定窗口的"最近 7 天"。
    """
    
    def __init__(self, max_context_tokens: int = 800):
        self.max_tokens = max_context_tokens
    
    def compress(
        self,
        current_diary: str,
        candidates: list[DiaryEntry],
        episodic_memory: list[dict],
    ) -> str:
        """
        策略：
        1. 对 candidates 按与 current_diary 的语义相似度排序
        2. 逐条加入上下文，直到接近 token 上限
        3. 优先加入 episodic_memory（AI 的记忆比原文更精炼）
        4. 对长日记进行摘要压缩而非截断
        """
        context_parts = []
        remaining_tokens = self.max_tokens
        
        # 优先注入 episodic memory（精炼、信息密度高）
        for mem in episodic_memory[:3]:
            snippet = f"[记忆] {mem['event']}（{mem['emotion']}）"
            tokens = self._estimate_tokens(snippet)
            if tokens <= remaining_tokens:
                context_parts.append(snippet)
                remaining_tokens -= tokens
        
        # 按相关性排序后注入历史日记
        ranked = self._rank_by_relevance(current_diary, candidates)
        for entry in ranked:
            summary = self._summarize_if_needed(entry, remaining_tokens)
            tokens = self._estimate_tokens(summary)
            if tokens <= remaining_tokens:
                context_parts.append(summary)
                remaining_tokens -= tokens
            else:
                break
        
        return "\n".join(context_parts)
```

```python
class IntentClassifier:

    使用规则 + 小模型混合方式判断是否需要 Agent 模式。
    避免硬编码关键词的脆弱性。

        返回:
        {
```

### 4.4 父子文档切分策略（mythink.txt 中提到的进阶优化）

```python
class ParentChildChunker:
    """
    检索时用小块（Child），返回时用大块（Parent）。
    解决"检索精度"和"上下文完整"之间的矛盾。
    """
    
    def chunk(self, diary_content: str, nid: int, date: str) -> dict:
        # Parent: 整篇日记（或按天聚合）
        parent = {
            "id": f"parent_{nid}",
            "content": diary_content,
            "date": date,
        }
        
        # Children: 200-300 字的语义聚焦小块
        children = self._split_to_children(diary_content, parent_id=f"parent_{nid}")
        
        return {"parent": parent, "children": children}
    
    def retrieve(self, query: str, user_id: int) -> list[str]:
        """在 children 中检索，返回对应的 parent 内容"""
        child_hits = self._search_children(query, user_id)
        parent_ids = set(c["parent_id"] for c in child_hits)
        return [self._get_parent(pid) for pid in parent_ids]
```

## 五、知识库建设

### 5.1 现状分析

当前知识库 = Chroma 向量库（按用户隔离的日记 Collection），仅存储日记原文。

**缺失：**
- 无结构化知识提取（人物、事件、地点的关系未建模）
- 无跨日记的知识聚合（同一话题分散在多篇日记中）
- 无外部知识注入（心理学知识、季节性情绪规律等）

### 5.2 三层知识库架构

```
┌──────────────────────────────────────────────────────┐
│  Layer C: Domain Knowledge (领域知识)                  │
│  来源：心理学常识、情绪管理方法论、正念技巧            │
│  存储：静态向量库（全局共享，只读）                    │
│  用途：AI 回应时引用专业知识，提升建议质量             │
├──────────────────────────────────────────────────────┤
│  Layer B: Structured Knowledge (结构化知识)            │
│  来源：从日记中自动抽取的实体和关系                    │
│  存储：JSON / 轻量图数据库                            │
│  用途：回答"我上次提到小明是什么时候"这类精确问题      │

│  Layer A: Raw Vectors (原始向量，已实现)               │
│  来源：日记原文 Chunk 化后的向量表示                   │

└──────────────────────────────────────────────────────┘
```

### 5.3 知识抽取 Pipeline

```python
class KnowledgeExtractor:
    """
    从日记中抽取结构化知识，构建用户个人知识图谱。
    使用 LLM 做 NER + 关系抽取（少量 Token 消耗换取长期价值）。
    """
    
    EXTRACTION_PROMPT = """从以下日记中提取关键信息，JSON格式输出：
{
  "people": [{"name": "xxx", "relation": "同事/朋友/家人", "sentiment": "正面/负面/中性"}],
  "events": [{"description": "xxx", "date": "推测日期", "emotion": "情绪标签"}],
  "places": ["地点1", "地点2"],
  "topics": ["主题1", "主题2"],
  "mood_score": 0.0-1.0
}

日记内容：{content}"""
    
    async def extract(self, content: str, llm) -> dict:
        # 仅对 > 100 字的日记做抽取（短日记信息密度低）
        if len(content) < 100:
            return {}
        response = await llm.ainvoke(self.EXTRACTION_PROMPT.format(content=content))
        return self._parse_json(response.content)
```

## 六、Skill 设计

### 6.1 现状分析

当前 Agent 的"技能"= 4 个 @tool 函数，功能固定，无法扩展。

### 6.2 Skill 注册表架构

将工具升级为可插拔的 Skill 系统：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from enum import Enum

class SkillCategory(Enum):
    RETRIEVAL = "retrieval"       # 信息检索类
    ANALYSIS = "analysis"         # 分析类
    GENERATION = "generation"     # 生成类
    EXTERNAL = "external"         # 外部服务类
    MEMORY = "memory"             # 记忆操作类

@dataclass
class SkillMetadata:
    name: str
    description: str
    category: SkillCategory
    token_cost_estimate: int      # 预估 token 消耗
    requires_db: bool             # 是否需要数据库
    requires_network: bool        # 是否需要网络
    priority: int                 # 优先级（Supervisor 选择时参考）

class BaseSkill(ABC):
    metadata: SkillMetadata
    
    @abstractmethod
    def execute(self, context: dict, **kwargs) -> str:
        """执行技能，返回结果文本"""
        pass
    
    @abstractmethod
    def should_activate(self, diary_content: str, intent: dict) -> float:
        """返回 0-1 的激活概率，Supervisor 据此决定是否调用"""
        pass
```

### 6.3 规划的 Skill 清单

| Skill 名称 | 类别 | 描述 | 触发条件 |
|------------|------|------|----------|
| `diary_semantic_search` | RETRIEVAL | RAG 语义检索（已有） | 含历史回溯意图 |
| `weather_query` | EXTERNAL | 天气查询（已有） | 含天气/出行相关 |
| `sentiment_analysis` | ANALYSIS | 情感分析（已有） | 长文 or 强烈情绪 |

### 6.4 Skill 动态加载与热插拔

```python
class SkillRegistry:
    """Skill 注册表，支持动态加载和优先级排序"""
    
    def __init__(self):
        self._skills: dict[str, BaseSkill] = {}
    
    def register(self, skill: BaseSkill):
        self._skills[skill.metadata.name] = skill
    
    def get_available_skills(
        self,
        diary_content: str,
        intent: dict,
        token_budget: int,
    ) -> list[BaseSkill]:
        """
        根据日记内容、意图和 Token 预算，
        返回应该激活的 Skill 列表（按优先级排序）。
        """
        candidates = []
        for skill in self._skills.values():
            activation_score = skill.should_activate(diary_content, intent)
            if activation_score > 0.3 and skill.metadata.token_cost_estimate <= token_budget:
                candidates.append((skill, activation_score))
        
        # 按激活分数 × 优先级排序
        candidates.sort(key=lambda x: x[1] * x[0].metadata.priority, reverse=True)
        
        # Token 预算内贪心选择
        selected = []
        remaining = token_budget
        for skill, score in candidates:
            if skill.metadata.token_cost_estimate <= remaining:
                selected.append(skill)
                remaining -= skill.metadata.token_cost_estimate
        
        return selected
```
