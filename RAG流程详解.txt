夜记管理系统 — 企业级 RAG 处理流程详解
========================================

本文档详细解释"夜记"项目中 RAG（Retrieval-Augmented Generation，检索增强生成）管线的完整处理流程。
RAG 的核心思想是：在让 LLM 生成回应之前，先从用户的历史数据中检索出最相关的内容作为上下文，
让 AI 的回应更加贴合用户的真实情况，而不是凭空生成。

本项目的 RAG 管线分为两大阶段：写入阶段（Indexing）和检索阶段（Retrieval）。


================================================================================
一、写入阶段（Indexing Pipeline）
================================================================================

写入阶段发生在用户创建或更新日记时。目的是将日记内容处理后存入向量数据库，
为后续检索做好准备。


**【1. Chunk 切分 — ChunkSplitter】**

当用户写完一篇日记并保存时，系统不会将整篇日记直接存入向量库，而是先进行文本切分。

为什么要切分？
- 一篇日记可能有几百甚至上千字，涵盖多个话题（早上的天气、中午的工作、晚上的心情）
- 如果整篇存入，检索时只能匹配到"整篇日记"，无法精确定位到最相关的那一段
- 切分后，每个 chunk 是一个独立的语义单元，检索时能精准匹配到最相关的段落

切分策略：
- 使用 LangChain 的 RecursiveCharacterTextSplitter，针对中文优化
- 分隔符优先级：段落换行 > 句号 > 感叹号 > 问号 > 分号 > 逗号 > 空格
- 每个 chunk 最大 512 个字符（CHUNK_SIZE 环境变量可配置）
- 相邻 chunk 之间保留 50 个字符的重叠（CHUNK_OVERLAP），防止跨段落的语义被截断
- 短日记（< 128 字符）不切分，直接作为单个 chunk 处理

举例：
  原始日记（600字）→ 切分为 2 个 chunk
  chunk_0: 前 512 字符（包含开头到某个句号）
  chunk_1: 从 chunk_0 尾部往前 50 字符开始，到日记结尾
  两个 chunk 之间有 50 字符的重叠区域，确保语义连贯

元数据：
  每个 chunk 会附加元数据：原始日记 ID（nid）、用户 ID（uid）、日期、标签、
  chunk 序号（chunk_index）和该日记的 chunk 总数（chunk_total）。
  这些元数据在检索时用于关联回原始日记。


**【2. 向量化 — Embedding Model】**

切分后的每个 chunk 需要转换为数学向量（一串浮点数），才能进行语义相似度计算。

Embedding 模型：
- 默认使用 shibing624/text2vec-base-chinese（中文优化，本地运行，免费）
- 支持通过 EMBEDDING_MODEL 环境变量切换为更强的模型（如 BAAI/bge-large-zh-v1.5）
- 输出 768 维的浮点数向量（EMBEDDING_DIMENSION 可配置）

工作原理：
  "今天心情很好" → [0.12, -0.34, 0.56, ..., 0.78]（768 个浮点数）
  "感到非常快乐" → [0.11, -0.33, 0.55, ..., 0.77]（768 个浮点数）
  这两个向量在 768 维空间中的距离很近（余弦相似度高），因为它们语义相近。
  而 "今天加班到很晚" 的向量则会离它们较远。

模型加载：
- 首次加载约需下载 400MB 模型文件，之后从本地缓存读取
- 设置离线模式（HF_HUB_OFFLINE=1），避免每次启动都联网检查
- 加载时记录模型名称和维度到日志，便于运维排查


**【3. 向量存储 — Chroma 向量数据库】**

向量化后的 chunk 存入 Chroma 向量数据库，这是一个专门为向量检索设计的数据库。

存储结构：
- 每个用户一个独立的 Collection（命名：user_{uid}_diary），实现数据隔离
- 每个 chunk 的 doc_id 格式：diary_{nid}_chunk_{chunk_index}
- 存储内容：chunk 文本 + 向量 + 元数据（nid, uid, date, tags, chunk_index, chunk_total）

持久化：
- 使用 PersistentClient 模式，数据存储在磁盘（chroma_data/ 目录）
- 服务重启后数据不丢失

写入时机：
- 创建日记：切分 → 向量化 → 批量写入所有 chunk
- 更新日记：先删除该日记的所有旧 chunk → 重新切分写入新 chunk
- 删除日记：通过 where={"nid": nid} 删除该日记的所有 chunk


**【4. BM25 索引构建】**

除了向量存储，系统还会在内存中为每个用户维护一个 BM25 倒排索引，用于关键词检索。

BM25 索引不是在写入时构建的，而是在首次检索时懒加载：
- 从 Chroma 中加载该用户的所有 chunk
- 使用 jieba 进行中文分词，过滤长度 <= 1 的停用词和标点
- 使用 rank_bm25 库的 BM25Okapi 算法构建倒排索引
- 索引缓存在内存中（_bm25_indexes 字典），后续检索直接使用
- 当用户的日记发生增删改时，自动失效该用户的 BM25 缓存，下次检索时重建


================================================================================
二、检索阶段（Retrieval Pipeline）
================================================================================

检索阶段发生在 AI 分析日记时。ReAct Agent 的 search_diary 工具会调用
search_similar_diaries 函数，触发完整的多路召回 + 重排序流程。


**【5. 语义检索 — Semantic Retriever】**

第一路召回：基于向量余弦相似度的语义检索。

流程：
1. 将用户的查询文本通过 Embedding 模型转换为 768 维向量
2. 在该用户的 Chroma Collection 中计算查询向量与所有 chunk 向量的余弦相似度
3. 返回相似度最高的前 SEMANTIC_TOP_K（默认 20）个 chunk

优势：
- 能理解语义，"开心" 能匹配到 "心情很好"、"感到快乐"
- 不依赖精确的关键词匹配

劣势：
- 对专有名词、数字、代码等精确匹配场景效果差
- "Python 3.11" 可能匹配不到 "Python 3.11"，因为向量空间中它们可能不够近


**【6. BM25 关键词检索 — BM25 Retriever】**

第二路召回：基于 BM25 算法的关键词检索，与语义检索互补。

流程：
1. 将查询文本通过 jieba 分词，过滤停用词
2. 在该用户的 BM25 索引中计算每个 chunk 的 BM25 分数
3. 返回分数最高的前 BM25_TOP_K（默认 20）个 chunk

BM25 算法简介：
- BM25 是信息检索领域的经典算法，基于词频（TF）和逆文档频率（IDF）
- 一个词在某个文档中出现越多（TF 高），且在整个语料库中出现越少（IDF 高），
  则该文档与查询的相关性越高
- 比简单的关键词匹配更智能，但不理解语义

优势：
- 精确匹配关键词，"Python" 就是 "Python"
- 对专有名词、人名、地名等效果好
- 计算速度快，不需要 GPU

劣势：
- 不理解语义，"开心" 匹配不到 "心情很好"

为什么需要两路召回？
- 语义检索和关键词检索各有盲区
- 语义检索可能漏掉精确关键词匹配的结果
- BM25 可能漏掉语义相近但用词不同的结果
- 两路互补，覆盖更全面


**【7. RRF 融合 — Reciprocal Rank Fusion】**

两路召回各返回 20 个结果，总共最多 40 个（有重叠的会更少）。
需要将它们融合为一个统一的排序列表。

RRF 算法：
- 公式：score(d) = Σ 1/(k + rank_i(d))
- 其中 k=60 是平滑常数，rank_i(d) 是文档 d 在第 i 路召回中的排名（从 1 开始）
- 如果一个文档同时被两路召回命中，它的 RRF 分数会更高（两个分数相加）

举例：
  文档 A 在语义检索中排第 1，在 BM25 中排第 3
  score(A) = 1/(60+1) + 1/(60+3) = 0.01639 + 0.01587 = 0.03226

  文档 B 只在语义检索中排第 2
  score(B) = 1/(60+2) = 0.01613

  文档 A 的 RRF 分数更高，排在前面。

为什么用 RRF 而不是简单合并？
- 不同检索方式的分数量纲不同（余弦距离 vs BM25 分数），不能直接比较
- RRF 只看排名不看分数，天然解决了量纲不一致的问题
- 算法简单高效，效果稳定，是业界广泛使用的融合方法

去重：
- 使用 doc_id 作为去重键，同一个 chunk 被两路都召回时只保留一份
- 但 RRF 分数会累加，体现"被多路召回"的优势


**【8. Re-rank 重排序 — CrossEncoder】**

RRF 融合后的结果虽然比单路召回好，但排序仍然是基于"排名融合"的粗排。
Re-rank 阶段使用更精确的模型对候选结果进行精排。

Re-rank 模型：
- 默认使用 BAAI/bge-reranker-base（CrossEncoder 架构）
- 与 Embedding 模型（Bi-Encoder）的区别：
  - Bi-Encoder：分别编码查询和文档，计算向量相似度，速度快但精度一般
  - CrossEncoder：将查询和文档拼接后一起输入模型，直接输出相关性分数，精度高但速度慢
- CrossEncoder 的精度更高，因为它能看到查询和文档之间的交互关系

流程：
1. 将 RRF 融合后的所有候选 chunk 与查询文本组成 (query, chunk) 对
2. CrossEncoder 模型对每一对计算相关性分数（rerank_score）
3. 按 rerank_score 降序排列
4. 截取前 RERANK_TOP_K（默认 5）个结果

为什么 Re-rank 放在最后？
- CrossEncoder 需要对每个候选逐一计算，计算量大
- 如果直接对所有 chunk 做 Re-rank，太慢
- 先用快速的向量检索和 BM25 缩小范围（20+20 → 融合后约 30 个），
  再用精确的 CrossEncoder 精排，是性能和精度的最佳平衡

降级策略：
- Re-rank 模型加载失败或执行异常时，降级返回 RRF 融合结果的前 top_k 个
- 不会因为 Re-rank 失败而导致整个检索链路中断


**【9. NID 去重聚合】**

Re-rank 后的结果是 chunk 级别的，但 AI Agent 需要的是日记级别的上下文。
同一篇日记可能有多个 chunk 被召回（比如 chunk_0 和 chunk_2 都很相关）。

去重逻辑：
- 按原始日记 nid 去重，同一 nid 的多个 chunk 只保留分数最高的那个
- 保持 Re-rank 的排序顺序
- 最终返回 top_k（默认 5）篇不同日记的最相关片段

返回格式：
  [{"nid": 42, "content": "chunk内容", "date": "2025-01-15", "tags": "#工作", "distance": 0.85}]

这个格式与升级前完全一致，确保 AI_Service 中的调用方无需修改。


================================================================================
三、生成阶段（Generation）
================================================================================


**【10. ReAct Agent 上下文注入】**

检索到的 5 篇最相关日记片段会作为上下文注入到 AI Agent 的 Prompt 中。

AI_Service 的 ReAct Agent 有一个 search_diary 工具，当 Agent 判断需要参考历史日记时，
会调用这个工具，触发上述完整的检索流程。检索结果会被格式化为：

  [2025-01-15] #工作 今天项目进展顺利，完成了核心模块的开发...
  [2025-01-10] #学习 学习了 RAG 的原理，理解了向量检索的工作方式...

这些历史上下文帮助 AI 更好地理解用户的处境，生成更贴合用户实际情况的温暖回应。


================================================================================
四、评估与调优
================================================================================


**【11. 召回率评估 — RecallEvaluator】**

系统提供了一个命令行评估工具，用于量化检索质量，帮助调优 RAG 管线。

运行方式：
  python -m scripts.evaluate_retrieval --user-id 1 --k 5

评估指标：
- Hit Rate@K：前 K 个结果中是否至少命中了一个期望结果（0 或 1）
- MRR@K（Mean Reciprocal Rank）：第一个命中结果的排名倒数的平均值
  - 如果第 1 个就命中，MRR=1.0；第 2 个命中，MRR=0.5；第 3 个命中，MRR=0.33
  - MRR 越高，说明相关结果排得越靠前
- Recall@K：前 K 个结果中命中的期望结果占期望总数的比例
  - 如果期望命中 3 篇日记，实际命中了 2 篇，Recall=0.67

四种模式对比：
  评估工具会分别输出以下四种模式的指标，便于对比每个环节带来的提升：
  1. 仅语义检索
  2. 仅 BM25 检索
  3. 混合检索（语义 + BM25 + RRF 融合）
  4. 混合检索 + Re-rank

  通常你会看到：混合检索 > 单路检索，混合+Re-rank > 混合检索。
  如果某种模式的指标反而更低，说明该环节的参数需要调优。


**【12. 数据迁移 — migrate_to_enterprise_rag.py】**

从旧版（整篇日记存入向量库）升级到新版（Chunk 模式）时，需要运行迁移脚本。

运行方式：
  python -m scripts.migrate_to_enterprise_rag

迁移流程：
1. 遍历所有用户的所有日记
2. 检查每篇日记是否已迁移（doc_id 中是否包含 "_chunk_"）
3. 删除旧的整篇向量（diary_{nid}）
4. 使用 ChunkSplitter 切分日记内容
5. 写入新的 chunk 向量（diary_{nid}_chunk_{0}, diary_{nid}_chunk_{1}, ...）
6. 支持增量迁移：已迁移的日记会被跳过
7. 异常时记录失败的日记 ID 并继续处理，最终输出迁移报告


================================================================================
五、完整数据流总结
================================================================================

写入流程：
  用户写日记 → ChunkSplitter 切分 → Embedding 向量化 → Chroma 存储 + BM25 缓存失效

检索流程：
  AI Agent 调用 search_diary
    → 语义检索取 20 个 chunk（SEMANTIC_TOP_K）
    → BM25 检索取 20 个 chunk（BM25_TOP_K）
    → RRF 融合去重排序（k=60 平滑常数）
    → Re-rank 精排取 5 个（RERANK_TOP_K）
    → NID 去重聚合
    → 返回 5 篇最相关日记片段给 AI Agent

生成流程：
  AI Agent 获取检索结果 → 注入 Prompt 上下文 → LLM 生成温暖回应

BM25_TOP_K=20                                        # BM25 检索召回数

CHUNK_OVERLAP=50                                     # Chunk 重叠字符数
